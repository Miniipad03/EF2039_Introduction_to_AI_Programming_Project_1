## 2. 버그 수정

### `dataset_utils.py` — val_test_ratio 오버플로우
- **문제**: `added` 이미지가 많아지면 비율이 1.0 초과 → `train_test_split` ValueError
- **수정**: `min(..., 0.90)` clamp 추가

```python
# Before
val_test_ratio = (0.4 * n_total) / n_orig
# After
val_test_ratio = min((0.4 * n_total) / n_orig, 0.90)
```

### `train.py` — Backbone LR Scheduler 미적용
- **문제**: `CosineAnnealingLR`은 초기화 시점 param group만 관리 → backbone을 `add_param_group`으로 추가해도 LR이 `lr * 0.1` 고정값으로 decay 안 됨. CBAM 실험의 과적합 원인 중 하나.
- **수정**: unfreeze 시점에 scheduler 재생성해서 backbone까지 Cosine decay 적용

```python
remaining = max(args.epochs - warmup_epochs, 1)
scheduler = optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=remaining)
```

## 3. Staged Fine-tuning 기법 논의

freeze → unfreeze 전략이 표준 기법인지에 대한 질문.

**결론: 표준 기법 맞음.** 이름은 Staged Fine-tuning (또는 Gradual Unfreezing).

**왜 쓰는가**:  
pretrained backbone은 ImageNet으로 잘 학습된 상태. 처음부터 전체를 같은 lr로 학습하면 randomly initialized fc head의 큰 gradient가 backbone까지 전파되어 기존 feature를 망가뜨림 (catastrophic forgetting).

1. **Warmup (epoch 1~5)**: backbone 동결, fc head만 학습 → head gradient 안정화
2. **Fine-tuning (epoch 6~)**: backbone 해제, 낮은 lr(1e-4)로 전체 학습

**보고서 작성 시**: 과제 명세 4.2절이 "frozen or fine-tuned, and why"를 명시적으로 요구하므로 이유를 한 줄 이상 써야 함. 별도 레퍼런스 인용 없이 설명만으로 충분.

**models.py 분리 이유**:  
Inference Demo, Grad-CAM 스크립트가 모델 구조를 import해야 함. `train.py` 안에 있으면 argparse, K-Fold 루프까지 딸려옴.

플롯 저장, CSV 저장 등도 `train.py`에 그대로 유지. `main()`이 순서대로 호출하는 구조이고 외부 재사용 없음. 547줄 단일 파일은 충분히 읽을 수 있는 수준.

## 9. Attention 메커니즘 논의 및 수정

### 삽입 위치 수정 (논문 방식으로 변경)

**기존 코드 (BlockWithAttention)**:
```
[conv1 → BN → ReLU → conv2 → BN → (+skip) → ReLU] → Attention
```
skip connection 결과까지 포함된 값에 attention 적용 → skip 보호 없음.

**수정 후 (BasicBlockWithAttention, SE-Net/CBAM 논문 방식)**:
```
conv1 → BN → ReLU → conv2 → BN → Attention → (+skip) → ReLU
```
residual path에만 적용, skip connection으로 gradient 직접 흐름.

### SE-Net Channel Attention vs 현재 ChannelAttention

| | SE-Net | 현재 코드 (CBAM 방식) |
|---|---|---|
| Pooling | avg만 | avg + max |
| MLP | 공유 아님 | 공유 |
| 방식 | squeeze → excitation | avg/max 두 신호 합산 |

현재 `ChannelAttention`은 SE-Net이 아닌 CBAM channel attention 구현.

### 삽입 레이어 결정 (layer3 + layer4 유지)

- shallow layer(1,2): 저수준 특징(엣지, 텍스처), attention 의미 적음
- deep layer(3,4): 고수준 semantic 특징, "어느 채널/위치가 중요한가" 판단 유효
- FGVC-Aircraft는 fine-grained 분류 → 고수준 feature 집중이 적합
- 모든 레이어 삽입은 비교 실험 목적상 불필요, 보고서 서술 근거로 활용 가능

### 수정 파일: `models.py`
- `BlockWithAttention` → `BasicBlockWithAttention` (논문 방식)
- `build_model` 내 잔여 참조 `BlockWithAttention` → `BasicBlockWithAttention` 버그 수정

## 11. GPU 자원 최적화

### 벤치마크 결과

| | 시간 |
|---|---|
| GPU 배치 처리 (batch=32, AMP) | 6 ms |
| DataLoader (workers=8) | 44 ms |
| DataLoader (workers=12) | 23 ms |
| DataLoader (workers=14) | 23 ms |

GPU가 7배 빠름 → GPU 가동률 약 12%. DataLoader가 병목.

### 분석
- workers=12 이상에서 포화 (14로 늘려도 효과 없음)
- 병목 원인: 코어 수가 아닌 ColorJitter 등 per-worker 연산 속도
- CPU 코어 추가 할당 → 효과 없음
- 근본 해결책은 batch_size 증가이나 소규모 데이터셋(~10k)에서 학습 영향 우려 → batch_size=32 유지
- 전체 실험 예상 시간: ~90분 (충분히 허용 범위)

### 적용된 수정 (`train.py`)

**AMP (Automatic Mixed Precision)**:
- `train_one_epoch`, `evaluate`, `ensemble_predict`에 `torch.amp.autocast('cuda')` 적용
- `GradScaler`로 backward 관리
- CPU 환경에서는 자동 비활성화

**num_workers**:
```python
# 전: 하드코딩 4
num_workers = 2 if os.name == 'nt' else max(os.cpu_count() - 2, 1)
# 현재 환경: 14
```
